# Акцентная рамка окна редактора + инструмент выбора её цвета

Дата: 2026-06-23
Статус: согласовано

## Проблема

Окно редактора snip визуально не отличается от остальных окон на рабочем столе. Среди множества тёмных окон (терминалы, браузер, IDE) его трудно быстро найти глазами.

## Решение

Добавить окну редактора постоянную акцентную рамку по периметру и инструмент выбора её цвета с большой палитрой. Цвет рамки сохраняется между запусками.

## Цели

- Окно редактора заметно выделяется среди прочих окон.
- Пользователь выбирает цвет рамки из большой палитры пресетов или задаёт произвольный.
- Выбранный цвет сохраняется и применяется при следующих запусках.

## Вне рамок (non-goals)

- Рамку нельзя выключить — она всегда видима.
- Окна capture не трогаем: захват области идёт через системный XDG-портал, своего окна у snip там нет.
- Толщина рамки не настраивается (фиксированная константа).

## Архитектура

Две точки изменений: `src/editor.rs` (рамка + UI инструмента) и `src/utils.rs` (конфиг).

### 1. Рамка окна

- На `AdwApplicationWindow` редактора добавляется CSS-класс `snip-bordered`.
- Отдельный `gtk::CssProvider` (назовём `border_css`) регистрируется на дисплее и задаёт правило:

  ```css
  window.snip-bordered { border: <WINDOW_BORDER_WIDTH>px solid rgba(r,g,b,a); }
  ```

- `border-radius` не переопределяется — рамка наследует скругление углов из темы и повторяет форму окна.
- Ширина рамки — константа `WINDOW_BORDER_WIDTH = 3`.
- Начальный цвет берётся из конфига при создании окна.

Риск: отрисовка `border` на узле `window` под Wayland/CSD (скругление + тень рисуются темой). Проверяется запуском приложения. Запасной вариант, если тема перебивает: рисовать рамку на корневом контейнере содержимого (`content`) вместо узла `window`. По умолчанию идём через CSS окна.

### 2. Инструмент выбора цвета рамки

Размещается в `HeaderBar` справа (`header.pack_end`).

- `gtk::MenuButton` с CSS-образцом текущего цвета рамки (по образцу кнопки цвета кисти из `build_bottom_bar`). Tooltip: «Цвет рамки окна».
- Образец красится через `color_button_css` + персональный `CssProvider` (`border_btn_css`), который обновляется при смене цвета.
- Поповер содержит:
  - **Сетку 8×5 = 40 пресетов** (`gtk::Grid`, 8 столбцов). Каждый пресет — `gtk::Button` с CSS-фоном через `color_button_css`. Клик: применяет цвет, закрывает поповер.
  - **Кнопку «Свой цвет…»** под сеткой. Открывает нативный `gtk::ColorDialog` (`choose_rgba`, доступен с GTK 4.10; в проекте GTK 4.16). Начальное значение — текущий цвет рамки. По результату применяет выбранный цвет.

```

ff5252 ffa726 ffe14d 66bb6a 26a69a 42a5f5 ab47bc f06292
```

### Применение выбранного цвета (общая процедура `apply_border_color`)

При выборе цвета (из пресета или из диалога) выполняется единая логика:

1. Перезагрузить `border_css` строкой с новым цветом → рамка окна перекрашивается вживую.
2. Перезагрузить `border_btn_css` → образец на кнопке обновляется.
3. Сохранить цвет в конфиг (`save_config`).
4. Закрыть поповер.

Состояние цвета рамки хранится в `EditorState` (новое поле `border_color: gdk::RGBA`), чтобы диалог «Свой цвет…» знал текущее значение и чтобы `apply_border_color` не дублировал источник истины.

### 3. Конфиг (`src/utils.rs`)

- Структуру `BrushConfig` переименовать в `Config` (теперь хранит больше, чем кисть).
- Добавить поле:

  ```rust
  #[serde(default = "default_border_color")]
  pub border_color: [f32; 4], // RGBA, дефолт #e8590c
  ```

  Атрибут `#[serde(default)]` гарантирует, что старые `config.json` без поля грузятся без ошибки.
- Функции `load_brush_config` / `save_brush_config` переименовать в `load_config` / `save_config`; обновить все вызовы в `editor.rs`.
- Поля кисти (`color`, `width`) остаются как есть — минимизируем churn.
- Дефолт `border_color` = `[0.910, 0.349, 0.047, 1.0]` (#e8590c).

## Обработка ошибок

- Сохранение конфига — best-effort (как сейчас: ошибки логируются, не падаем).
- `ColorDialog` может быть отменён пользователем — тогда цвет не меняется (ветка ошибки/отмены игнорируется молча).
- Загрузка CSS — `load_from_string` не паникует; некорректная строка просто не применится (строки формируем сами, риск минимален).

## Тестирование

- Юнит-тест в `utils.rs`: `Config` с заданным `border_color` сериализуется и десериализуется без потерь.
- Юнит-тест: JSON старого формата (только `color` + `width`, без `border_color`) десериализуется в `Config` с дефолтным `border_color`.
- Ручная проверка запуском (`cargo run -- edit <path>`): рамка видна, скругление корректно, выбор пресета и «Свой цвет…» перекрашивают окно вживую, цвет переживает перезапуск.

## Затрагиваемые файлы

- `src/editor.rs` — рамка окна, кнопка инструмента в HeaderBar, поповер с палитрой, проводка `apply_border_color`, чтение цвета при старте.
- `src/utils.rs` — `Config`, `border_color`, `load_config`/`save_config`, тесты.
